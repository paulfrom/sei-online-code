## #5 codex home 符号链接树

**`CodexSandboxConfig.java`** — 新增 `linkSharedHome(Path codexHome, Logger logger)` + 包级重载 `(codexHome, userHome, logger)`（供测试注入临时家目录，同 `seedUserSkills` 模式）：
- 解析 shared home：`System.getenv("CODEX_HOME")`（JVM env，非 ProcessBuilder 注入的 per-run）→ 缺省 `userHome/.codex`。
- 符号链接（共享，multica `ensureSymlink`/`ensureDirSymlink`/`exposeSharedCodexPluginCache` 语义）：
  - `auth.json`（文件）：源缺失 no-op；dst 已是正确符号链接 no-op；dst 异常→删后重建。
  - `sessions/`（目录）：源不存在则先 `mkdir`（codex 会写），再链接。
  - `plugins/cache`（目录）：同 sessions（multica `exposeSharedCodexPluginCache`）。
- 隔离拷贝（multica `syncCopiedFile` 语义——每次刷新，非「不覆盖」）：`instructions.md`、`config.json`；源缺失则删 dst（不残留 stale 拷贝）。
- **不动 `config.toml`**（已由 `write` 托管；plan #5 明确只拷 instructions.md/config.json）。
- 符号链接用 `Files.createSymbolicLink`；失败（UnsupportedOperationException/IOException）→ 文件回退拷贝、目录 warn no-op（Linux 生产路径不受影响，测试机兼容）。
- **`CodexRunner.runBlocking`**：`write` 后调用 `linkSharedHome(codexHome, LOGGER)`（顺序：write → linkSharedHome → seedUserSkills → writeMcpBlock，各触不同文件）。

**测试**（`CodexSandboxConfigTest` 扩展）：auth.json 符号链接、sessions/ 符号链接、plugins/cache 符号链接、instructions.md 拷贝、源缺失 no-op、dst 错链重建。

## #6 MCP 托管块（全链路）

**后端实体/DTO/迁移**（镜像 cliTool PR1）：
- `Agent.java`：`@Column(name = "mcp_config", columnDefinition = "TEXT") private String mcpConfig;` + getter/setter。（null = 不托管，沿用 CLI 默认；`{}` = 托管空集，strict。）
- `AgentDto.java`：`@Schema(description = "MCP server 配置 JSON（Claude 风格 {\"mcpServers\":{...}}）；空表示不托管") private String mcpConfig;` + getter/setter。（sei-core `BaseEntityController` 按字段名自动映射，无需改 mapper——同 cliTool。）
- `V13__agent_mcp_config.sql`：`ALTER TABLE oc_agent ADD COLUMN mcp_config TEXT;`

**Runner 写入器**（`CodexSandboxConfig.java`，集中 config.toml 操作）— 移植 multica：
- `writeMcpBlock(Path codexHome, String mcpConfig, Logger logger)`：读 config.toml → 剥既有 MCP 托管块 → 若 `hasManagedCodexMcpConfig`（非 null/非空白 JSON）则剥用户 `[mcp_servers.*]` 表 + 渲染新块追加 → 写回 + `chmod 0o600`（secrets 走 `mcp_servers.<id>.env`，不入 argv，对齐 multica 安全论证）。
- 私有助手：`renderCodexMcpServersBlock`（Claude `{mcpServers}` → TOML `[mcp_servers.<name>]`，键名排序，bare-key 校验，http server 归一化 `headers`→`http_headers` + `experimental_use_rmcp_client`）、`stripCodexUserMcpServerTables`、`jsonValueToCodexTOMLInline`、`codexTOMLBasicString`、`codexTOMLKey`、`isCodexBareTomlKey`、`hasManagedCodexMcpConfig`。
- Markers：`# BEGIN sei-managed mcp_servers (do not edit; regenerated by CodexSandboxConfig)` / `# END sei-managed mcp_servers`。
- `mcpConfig` null/blank → 仅剥既有块，不写新块（CLI 默认回退）。

**签名 + 接线**：
- `CliRunner.java`：两 `execute` 重载各加 `String mcpConfig` 尾参（model 之后）。
- `ClaudeRunner`：接受 mcpConfig，**忽略**（TODO(oma-deferred)：claude MCP 走 `--mcp-config` 标志，另立设计；当前 claude 不托管 MCP，对齐 plan #6 仅指 codex）。

- mock arity 同步：`PlanAgentServiceTest`、`FeatureDesignBuildServiceTest`、`ClaudeRunnerRealClaudeTest`、`CodexRunnerRealCodexTest`、`CodexRunnerFakeExecutableTest`。

**测试**：`CodexSandboxConfigTest` 加 MCP 用例（stdio server 渲染、http server 归一化、剥用户 mcp 表、幂等、0o600、`{}` 空集写空 marker 块、null/blank 仅剥、bare-key 非法名拒收）；`CodexRunnerFakeExecutableTest` 加：传 mcpConfig 时 config.toml 含 `[mcp_servers.<name>]`。
